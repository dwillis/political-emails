# Dashboard Design — political-emails

**Date:** 2026-04-17
**Status:** Approved design, ready for implementation

## Context

The political-emails repo currently publishes a download-focused `docs/index.html`
via GitHub Pages (serving from `/docs` on main). The page lists JSONL
archives by year with a simple 3-stat bar at the top. This design replaces
that index with a richer dashboard home page and moves the full
downloads archive to a secondary page.

## Goals

- Show the extent of the collection at a glance
- Communicate the D/R party split and how it has evolved over time
- Surface the most-active sender domains
- Keep the downloads accessible but de-emphasized on the landing page
- Static build: no JavaScript, no runtime chart library

## Non-Goals

- Interactivity (hover tooltips, filtering, drilldown)
- Search / query UI
- Per-sender or per-email pages
- Historical diffs / changelog

## Architecture

Two pages live in `docs/`, both generated by `scripts/build_site.py`:

### `index.html` — Dashboard (home page)

```
┌─ Header (forest green bar, title, GitHub + Downloads links)
├─ Stats row — 4 cards:
│    Total Emails | With Disclaimer | Democratic | Republican
├─ Chart 1: "Emails per year" (vertical bar, amber)
├─ Chart 2: "Party breakdown by year" (stacked bar, blue/red/gray)
├─ Chart 3: "Top 10 sender domains" (horizontal bar, amber)
├─ Compact downloads:
│    Current month ZIP (prominent)
│    Latest complete year ZIP (prominent)
│    → Link to downloads.html
└─ Footer
```

### `downloads.html` — Full archive

The existing collapsible year-by-year download listing (current
`index.html` content, structurally unchanged). Header has a
"← Back to dashboard" link.

## Visual design

- **Primary**: `#1e4d2b` (deep forest green) — header, headings, borders
- **Accent**: `#e89b3c` (warm amber) — links, bar charts, hover states
- **Light accent**: `#f4c37d` (pale amber) — subtle highlights
- **Party colors** (charts only): D = `#2b6cb0`, R = `#c53030`,
  Unknown = `#a0aec0`
- Typography: Libre Baskerville for headings, system sans for body
- Layout: max-width 800px, responsive

## Data flow

Single-pass scan over every JSONL file in `data/` accumulates
counters in a `Stats` dict:

```python
{
    "total_records": int,
    "disclaimer_count": int,
    "party_counts": {"D": int, "R": int, "unknown": int},
    "unique_domains": int,
    "by_year": {
        "2019": {"total": int, "D": int, "R": int,
                 "unknown": int, "disclaimer": int},
        ...
    },
    "top_domains": [("actblue.com", 12345), ...],  # top 10
}
```

All counters bounded: 12 years × 4 buckets + ~4K unique domains.
Single pass fuses the current `compute_stats()` + download scanning
work, so the new build is actually faster than the old one.

Download metadata (ZIP sizes, per-month breakdowns) continues to
come from `build_downloads()`, unchanged.

## Charts (static SVG)

All charts generated as inline SVG strings by a new `scripts/charts.py`
module. No JS. Responsive via `viewBox="0 0 800 400"`.

**Shared conventions:**
- 60px left margin (y-axis labels), 40px bottom margin (x-axis labels)
- Grid lines and axis labels in muted gray (`#999`)
- `role="img"` + `aria-label` for accessibility
- Value labels drawn on/near bars

**Chart 1 — Emails per year** (vertical bar):
- x-axis: years, y-axis: count
- Amber bars, light grid lines at 25/50/75/100% of max

**Chart 2 — Party breakdown by year** (stacked bar):
- Same axes as Chart 1
- Stacked: D (blue, bottom) → R (red, middle) → Unknown (gray, top)
- Legend above chart
- Percentage labels on segments ≥ 8% of bar

**Chart 3 — Top 10 sender domains** (horizontal bar):
- y-axis: domain names (descending, biggest at top)
- x-axis: count
- Amber bars, 24px tall, 8px gap, value label at end

**Helper signatures:**
```python
def vertical_bar_chart(data: dict[str, int], title: str, color: str) -> str: ...
def stacked_bar_chart(data: dict[str, dict[str, int]],
                      categories: list, colors: dict, title: str) -> str: ...
def horizontal_bar_chart(data: list[tuple[str, int]],
                         title: str, color: str) -> str: ...
```

Each returns an SVG string. `build_site.py` imports and embeds.

## Testing

- **`charts.py`**: unit tests assert expected SVG elements for known
  inputs. Pure functions, deterministic output.
- **`build_site.py`** end-to-end: run against committed `data/`,
  confirm both HTML files produced, visually verify in browser.

## Error handling

- Missing/empty `data/` — print warning, write minimal placeholder HTML.
- JSONL parse errors — skip malformed lines, log count.
- Missing/null `party` or `domain` fields — bucket under "unknown",
  skip for top-domain aggregation.
